# StockTracker — Accessibility (a11y) Audit

**Date:** 2026-07-04
**Scope:** All 10 routes and ~50 UI components.
**Conformance target:** WCAG 2.2 AA.
**Status:** All findings across **P0 (critical)**, **P1 (serious)**, **P2 (moderate)**, and
**P3 (minor/verify)** addressed on 2026-07-04. The WCAG-2.2 items in §5 still warrant a runtime
axe/Lighthouse + screen-reader pass.

---

## 1. Summary & methodology

Every route under `src/app/**` and every component under `src/components/**` was reviewed for
accessibility gaps, with the highest-impact findings spot-verified against source
(`company-tabs.tsx`, `companies-table.tsx`, `authenticated-shell.tsx`, `layout.tsx`).

The shared primitives in `src/components/ui/*` are built on **base-ui** (not Radix). base-ui
provides solid built-in accessibility — focus trapping, focus restoration, roving `tabindex`,
`aria-modal`, and the correct roles — **but only when those primitives are actually used.** The
gaps found cluster in three categories:

1. **Hand-rolled interactive widgets** that bypass the accessible primitives (tabs, combobox,
   clickable rows).
2. **Icon-only controls** rendered without an accessible name.
3. **Color-only** encoding of information (profit/loss, allocation status).

Counts: **3 critical** (✅ all fixed), **5 serious** (✅ all fixed), **6 moderate** (✅ all fixed),
**3 minor/verify** (✅ all addressed), plus **4 WCAG-2.2 criteria** requiring a runtime pass to confirm.

The `lang="en"` attribute (`app/layout.tsx:29`) and the `<main>` landmark
(`authenticated-shell.tsx:50`) are present — good baselines to build on.

## 3. Detailed findings

### P0 — Critical (keyboard / operability blockers) — ✅ ALL FIXED (2026-07-04)

**1. Clickable table rows are not keyboard-operable** — `dashboard/companies-table.tsx:707`
*WCAG 2.1.1 Keyboard (A), 4.1.2 Name/Role/Value (A).*
✅ **Fixed:** the company-name cell now contains a real focusable
`<Link href="/company/{id}">` with a `focus-visible` ring. Keyboard and AT users can Tab to it and
press Enter; the row-level `onClick` is retained as a mouse convenience (the link
`stopPropagation`s to avoid a double navigation). This also gives middle-click / "open in new tab"
for free.

**2. Company detail tabs are hand-rolled, not a real tab widget** — `company/company-tabs.tsx`
*WCAG 4.1.2 (A), 2.1.1 (A).*
✅ **Fixed:** migrated from `<button>`-toggled `hidden` divs to the base-ui `Tabs` primitive
(`Tabs.Root` / `Tabs.List` / `Tabs.Tab` / `Tabs.Panel`), which supplies
`role="tablist"/"tab"/"tabpanel"`, `aria-selected`, `aria-controls`/`aria-labelledby`, and
Left/Right/Home/End arrow-key navigation. Panels use `keepMounted` to preserve the previous
always-mounted behaviour (tab-local form state survives switching). The underline styling is
preserved via `data-active:` variants, and a `focus-visible` ring was added.

**3. Stock search is a hand-rolled combobox without ARIA** — `company/stock-search.tsx`
*WCAG 4.1.2 (A), 2.1.1 (A).*
✅ **Fixed:** the input now exposes `role="combobox"`, `aria-expanded`, `aria-controls`,
`aria-autocomplete="list"`, and `aria-activedescendant`; the results list is `role="listbox"` with
`role="option"` + `aria-selected` items. Added full keyboard navigation (ArrowUp/Down, Home/End,
Enter to select, Escape to close) with the active option scrolled into view, following the
focus-stays-in-input activedescendant pattern.

### P1 — Serious — ✅ ALL FIXED (2026-07-04)

**4. Icon-only controls with no accessible name** — each needs an `aria-label`.
*WCAG 4.1.2 (A), 2.4.4 (A).*
✅ **Fixed:** added `aria-label` to the icon-only controls and `aria-hidden="true"` to the glyphs:
- `auth/user-nav.tsx` — avatar/initials menu trigger (`aria-label="Account menu"`).
- `import/page.tsx` — remove-file `XCircle` (`Remove {file}`) and delete-history `Trash2` buttons.
- `account/accounts-manager.tsx` — `Pencil` / `Trash2` / `Check` / `X` per account.
- `company/holdings-tab.tsx` — per-row edit / remove / save / cancel buttons.
- `portfolio/create-portfolio-dialog.tsx` — 8 color-swatch buttons (`aria-label` + `aria-pressed`).
- `ui/rich-text-editor-impl.tsx` — ColorPicker swatches (`aria-label` + `aria-pressed`) and the
  hidden image/PDF file inputs.
- `ui/select.tsx` — scroll up/down arrows are decorative, so their chevrons are `aria-hidden`.
- `dashboard/delete-company-dialogs.tsx` — **no change needed**: both trash triggers already carry
  visible text ("Delete" / "Delete All"), so they are not icon-only.

**5. Rich-text toolbar uses `title` instead of `aria-label`** —
`ui/rich-text-editor-impl.tsx`
`ToolbarButton`, `FontSizeSelect`, and `TableMenu` relied on `title`, which is not reliably
announced by screen readers. *WCAG 4.1.2 (A).*
✅ **Fixed:** the toolbar wrapper is now `role="toolbar"` with `aria-label="Text formatting"`; every
`ToolbarButton` exposes `aria-label` (from its `title`), `aria-pressed` (active state), and
`aria-disabled`; `FontSizeSelect` gained `aria-label="Font size"`. `TableMenu` items already carry
visible text labels.

**6. Color-only encoding of financial status** — meaning conveyed by red/green alone.
*WCAG 1.4.1 Use of Color (A).*
✅ **Fixed:**
- `dashboard/portfolio-pnl-bar.tsx` — already rendered a `+`/`-` sign; added an `aria-label`
  ("Profit: …" / "Loss: …") and `aria-hidden` on the visual value to avoid double announcement.
- `dashboard/allocation-summary-bar.tsx` — each chip now has an `aria-label` stating the star tier,
  percentage, target range, and an explicit status word ("under target" / "in range" /
  "over target"); the color-coded inner spans are `aria-hidden`.
- `dashboard/companies-table.tsx` — the P&L % and ₹ cells already carry explicit `+`/`-` signs
  (a non-color indicator); the allocation left-border in the portfolio view now also has a `title`
  and an `sr-only` "allocation {status}" text in the name cell. The allocation view already shows a
  textual Status column.

**7. Data tables missing `scope` on header cells** — `ui/table.tsx` `TableHead` (the base for all
tables), plus `company/holdings-tab.tsx`, `company/projection-grid.tsx`.
*WCAG 1.3.1 Info & Relationships (A).*
✅ **Fixed:** `TableHead` now defaults to `scope="col"` (overridable); `holdings-tab` headers are
`scope="col"` and the per-row account cell is a `<th scope="row">`; `projection-grid` year headers
are `scope="col"` and the row-label cells are now `<th scope="row">`. `company/valuation-scenarios.tsx`
is a **CSS-grid layout, not a real `<table>`**, so `scope` does not apply there.

**8. Star ratings have no text alternative** — `dashboard/companies-table.tsx`
(`"★".repeat(n)`) and elsewhere. Screen readers read a run of star glyphs with no rating context.
*WCAG 1.1.1 (A) / 1.3.1 (A).*
✅ **Fixed:** star runs are wrapped with `aria-label="{n} of 4 stars"` (the app uses a 1–4 scale)
and the glyphs are `aria-hidden`. Applied via a shared `StarRating` helper in `companies-table.tsx`
(3 spots) and inline in `dashboard/allocation-summary-bar.tsx` and `company/company-header.tsx`.

### P2 — Moderate — ✅ ALL FIXED (2026-07-04)

**9. No skip-to-content link** — `app/layout.tsx` / `authenticated-shell.tsx`. *WCAG 2.4.1 (A).*
✅ **Fixed:** added a visually-hidden "Skip to content" link (`sr-only focus:not-sr-only`) as the
first focusable element in `authenticated-shell.tsx`, targeting `#main-content`; the `<main>` now
has `id="main-content"` and `tabIndex={-1}` so focus lands there.

**10. Navigation missing current-page indication** — `layout/app-header.tsx`. *WCAG 4.1.2 (A).*
✅ **Fixed:** nav links are derived from `pathname`; the active link gets `aria-current="page"` plus
the emphasized text style, and the `<nav>` gained `aria-label="Primary"`.

**11. Async status not announced** — *WCAG 4.1.3 Status Messages (AA).*
✅ **Fixed:** added `role="status"` + `aria-live="polite"` to the dashboard "Loading companies…"
state, the import "Importing…" progress, and the accounts-manager / holdings-tab loading states;
spinner icons in those regions are `aria-hidden`. Sonner already emits an `aria-live` region, and
the P&L / allocation bars intentionally stay silent (a live region there would announce on every
price refresh).

**12. Form-labelling gaps** — *WCAG 1.3.1 (A), 3.3.2 (A), 4.1.2 (A).*
✅ **Fixed:**
- `company/company-form.tsx` — `StockSearch` now takes an `inputId` forwarded to its input, and the
  `<Label htmlFor="stock-search">` associates with it; the Position card's Account/Quantity/Avg-price
  fields got matching `htmlFor`/`id` pairs.
- `account/account-select.tsx` — the native `<select>` accepts an `id` and has `aria-label="Account"`;
  the inline new-account `Input` gained `aria-label="New account name"`.
- `settings/allocation-ranges-editor.tsx` — the min/max inputs have explicit `aria-label`s ("{n} star
  minimum/maximum allocation percent"); the `★` glyph is `aria-hidden` with an `aria-label` on its span.
- `company/holdings-tab.tsx` — the add-holding group is now a `<fieldset>` with a `<legend>` and
  `htmlFor`/`id`-associated labels.
- Verified `ui/input.tsx` + `ui/label.tsx` spread props, so `id`/`htmlFor` forward correctly.

**13. Per-page titles not set** — *WCAG 2.4.2 Page Titled (A).*
✅ **Fixed:** the root layout now defines a `title` template (`%s · StockTracker`). Server pages
export `metadata` (login → "Sign in", settings → "Settings") or `generateMetadata` (company detail →
the company name). Client pages that can't export `metadata` get a server `layout.tsx` wrapper
(import → "Import Holdings", new-company → "Add Company"); the dashboard sets `document.title`
from the selected portfolio in an effect.

**14. Card titles are `<div>`, not headings** — `ui/card.tsx`. *WCAG 1.3.1 (A).*
✅ **Fixed:** `CardTitle` now renders a semantic `<h3>` by default and accepts an `as` prop to
override (e.g. `as="h2"`) where the heading outline needs it. Overall heading order (single `h1`
per page, no large skips) should still be spot-checked in the runtime pass.

### P3 — Minor / verify — ✅ ALL ADDRESSED (2026-07-04)

**15. Decorative elements not hidden from AT** — `ui/tooltip.tsx`, `ui/separator.tsx`. *WCAG 1.3.1 (A).*
✅ **Fixed / verified:** the tooltip `Arrow` is now `aria-hidden="true"`. base-ui's `Separator`
renders a valid `role="separator"` element (a legitimate ARIA role, not announced as content), so no
change was needed there.

**16. `aria-modal` + focus management on dialogs** — `ui/dialog.tsx`, `ui/alert-dialog.tsx`,
`ui/sheet.tsx`. *WCAG 4.1.2 (A), 2.4.3 (A).*
✅ **Verified:** base-ui's `Dialog`/`AlertDialog` are modal by default — the popup carries
`role="dialog"` + `aria-modal`, is labelled by `DialogTitle`, and base-ui traps focus while open and
restores it to the trigger on close. The close button's `XIcon` was also marked `aria-hidden` (its
`sr-only` "Close" text supplies the name). Still worth a quick VoiceOver confirmation at runtime.

**17. `ui/skeleton.tsx`** — *WCAG 4.1.3 (AA).*
✅ **Fixed:** rather than annotate every skeleton, the sole consumer
(`company/[id]/loading.tsx`) now wraps the placeholders in a single `role="status"`
`aria-live="polite" aria-busy="true"` region with an `sr-only` "Loading company details…" message.

## 5. Requires manual / runtime verification

These cannot be confirmed by static reading alone and warrant an axe/Lighthouse pass plus a
keyboard/screen-reader walkthrough:

- **Findings 15–17 (code changes landed 2026-07-04)** — re-confirm at runtime: tooltip arrow /
  separator are silent, dialog `aria-modal` + focus trap/restore behave, and the skeleton
  `role="status"` announces once. Also confirm the new heading outline (finding 14, `CardTitle` →
  `<h3>`) has no jarring level skips per page.
- **2.4.7 Focus Visible (AA) / 2.4.11 Focus Not Obscured (AA, new in 2.2) / 2.4.13 Focus Appearance
  (AAA).** Grep for `outline-none` / `focus:outline-none` without a `focus-visible` ring; confirm
  the sticky header does not obscure focused elements on scroll.
- **2.5.8 Target Size (Minimum) 24×24 px (AA, new in 2.2).** The icon buttons and the 8 color
  swatches in create-portfolio / rich-text ColorPicker are likely under 24 px — measure.
- **1.4.3 Contrast (Minimum) (AA).** `--muted-foreground` (used pervasively) and
  `bg-destructive/10` + `text-destructive` in dark mode are both suspect — run a contrast checker.
- **3.3.8 Accessible Authentication (AA, new in 2.2).** Google-OAuth-only login has no cognitive
  puzzle, so it likely passes; confirm the login control is a real labelled `<button>`/`<a>` with
  text.
